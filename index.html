📋 Module 01 · Cloud & AWS 101 · ⏱ ~20 min
Module 01
☁️ Why Cloud? Why AWS?
Before touching S3, let's build your mental model of what cloud computing actually means.

The Old World vs The Cloud World

🧠 Real-World Analogy

Imagine you need electricity. You could build your own power plant (on-premise servers) — buy generators, hire engineers, maintain fuel. OR you could just plug into the power grid (cloud) — pay only for what you use, no maintenance headaches. That's exactly what AWS is: a global power grid for computing.

📋 Module 02 · S3 Fundamentals · ⏱ ~25 min
Module 02
🗄️ What IS Amazon S3?
🧠 The Perfect Analogy

S3 = An infinite Google Drive for developers. Imagine a magic filing cabinet that never runs out of space, never loses a file, is accessible from anywhere in the world via the internet, and charges you only for what you actually store. That's S3. The "3" in S3 stands for Simple Storage Service.

S3's Core Promise

📊 Diagram 3 — S3 Core Properties

What Can You Store in S3?

S3 is object storage — it stores any file as a "blob" without caring about its structure. Think of it as storing files in a flat folder (no nested directories, just key-value).

📷
Images & Videos

Profile pics, thumbnails

📄
Documents

PDFs, Word files, CSVs

🌐
Static Websites

HTML/CSS/JS files

💾
Backups

DB dumps, archives

📊
Big Data

Logs, data lake files

🤖
ML Datasets

Training data, models

⚠️ S3 is NOT a File System

S3 is object storage, not block storage (that's EBS) or file storage (that's EFS). You can't "mount" S3 like a hard drive or run a database directly on it. It's designed for storing and retrieving files via API calls. Objects can be 0 bytes to 5 TB each.

📋 Module 03 · Buckets & Objects · ⏱ ~30 min
Module 03
🪣 Buckets & Objects Deep Dive
🧠 Analogy

A Bucket is like a labeled box you own. An Object is each item inside that box. The Key is the unique sticker/label on each item. Every item in every box is uniquely findable via: box-name / item-label

📊 Diagram 4 — Bucket → Object Hierarchy

Bucket Rules You Must Know

  • Bucket names are globally unique across all AWS accounts worldwide (like domain names)
  • Names must be 3–63 characters, lowercase, no underscores
  • Buckets live in a specific region — data physically stays there (unless you replicate it)
  • One account can have up to 100 buckets by default (can be increased)
  • Unlimited objects per bucket — no ceiling

Versioning — The Time Machine

By default, uploading a file with the same name overwrites the old one. Enable Versioning to keep every historical version.

📊 Diagram 5 — S3 Versioning

You can set rules that automatically move or delete objects based on age — like a robot janitor for your storage.

📋 Module 04 · Security & Access · ⏱ ~30 min
Module 04
🔐 Security & Access Control
🧠 Security Analogy

Think of S3 security like a bank vault system. The bank (AWS) has master keys. You (the account owner) have a vault key. You can give your employees (IAM users) partial keys. You can also put a lock-combination rule on the vault itself (bucket policy). And you can issue a temporary pass for a visitor (pre-signed URL).

📊 Diagram 7 — S3 Security Layers (Defense in Depth)

IAM Policies — The Rulebook

IAM Policies are written in JSON. Here's what a typical S3 read-only policy looks like:

// IAM Policy: Allow read-only access to a specific S3 bucket { "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", // Allow or Deny "Action": [ "s3:GetObject", // Can download files "s3:ListBucket" // Can list files ], "Resource": [ "arn:aws:s3:::my-company-assets", // The bucket "arn:aws:s3:::my-company-assets/*" // All objects inside ] } ] }
📊 Diagram 8 — Public vs Private + Pre-Signed URLs
✅ Golden Rule of S3 Security

Start with everything private (default). Only open what you need. Use bucket policies over ACLs. Use IAM roles (not access keys) for your applications. Enable S3 Block Public Access at the account level — it's a safety net that prevents accidental public exposure.

📋 Module 05 · Storage Classes · ⏱ ~25 min
Module 05
💰 Storage Classes — Pay Smart
🧠 Analogy

Think of storage classes like hotel booking tiers. A suite (Standard) is instant, always ready, expensive. A standard room (Standard-IA) is available but you pay extra to check in. A hostel dorm (Glacier) takes 5 mins to get your key. A storage unit outside town (Deep Archive) takes 12 hours — but it's dirt cheap. Pick based on how fast you need the data.

📊 Diagram 9 — S3 Storage Class Comparison
💡 Which Class Should I Use?
  • Building an app? → Standard
  • Storing logs you check monthly? → Standard-IA
  • Not sure how often data is accessed? → Intelligent-Tiering
  • Long-term backups you rarely need? → Glacier Instant Retrieval
  • 7-year compliance archives? → Glacier Deep Archive
📋 Module 06 · Hands-On Lab · ⏱ ~40 min
Module 06 — Hands-On Lab
🛠️ Your First S3 Setup

We'll create a bucket, upload a file, control access, and host a static website. Follow the steps in the AWS Console (free tier applies).

📊 Diagram 10 — What We're Building

Step-by-Step Lab

  • 1
    Sign in to AWS Console

    Go to console.aws.amazon.com. Search "S3" in the services search bar and open it.

  • 2
    Create a Bucket

    Click "Create bucket" → Enter a globally unique name like your-name-learning-s3-2026 → Choose region us-east-1 → Leave all other defaults → Click "Create bucket".

  • 3
    Upload Your First Object

    Click your bucket → "Upload" → "Add files" → Pick any image or text file from your computer → Click "Upload". That's it! Your file is now in S3.

  • 4
    Enable Static Website Hosting

    Go to bucket → "Properties" tab → Scroll to "Static website hosting" → Enable it → Set index document to index.html → Save.

  • 5
    Create & Upload index.html

    Create a file called index.html on your laptop with the code below, then upload it to your bucket.

<!-- Save this as index.html and upload to your S3 bucket --> <!DOCTYPE html> <html> <head> <title>My First S3 Website 🚀</title> <style> body { font-family: Arial; text-align: center; background: #232F3E; color: white; padding: 60px; } h1 { color: #FF9900; font-size: 3rem; } </style> </head> <body> <h1>☁️ Hello from S3!</h1> <p>I built this with Amazon S3 Static Website Hosting</p> <p>Bucket: your-bucket-name · Region: us-east-1</p> </body> </html>
  • 6
    Add a Bucket Policy to Make it Public

    Go to "Permissions" tab → First, turn off "Block all public access" → Then go to "Bucket policy" and paste the JSON below (replace your-bucket-name):

{ "Version": "2012-10-17", "Statement": [ { "Sid": "PublicReadGetObject", "Effect": "Allow", "Principal": "*", // * = anyone in the world "Action": "s3:GetObject", "Resource": "arn:aws:s3:::YOUR-BUCKET-NAME/*" } ] }
  • 7
    Visit Your Website!

    Go back to "Properties" → "Static website hosting" → Click the website endpoint URL. Your S3-hosted website is live! 🎉

🎉 Congratulations!

You just created a bucket, uploaded objects, configured permissions, and hosted a website — all on AWS S3! When you're done, delete the bucket to avoid any charges (S3 free tier gives 5GB/month but it's good practice).
